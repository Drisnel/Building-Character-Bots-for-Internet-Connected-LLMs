# Workflow des Introductions et Démarrages Canons — Comment je Construis les Ouvertures RP pour des Bots Canons

Après avoir créé et testé un grand nombre de personnages canons, j'ai fini par réaliser qu'un bon script ne suffit pas.

L'introduction et le démarrage peuvent renforcer le script.

Ils peuvent également le fragiliser.

Sur certaines plateformes, les introductions et les démarrages existent indépendamment du script du personnage.

Sur d'autres, en particulier les plateformes connectées à Internet, le démarrage peut influencer directement le comportement futur.

Pour cette raison, je ne considère pas les introductions et les démarrages comme séparés du personnage.

Je considère :

- le script
- l'introduction
- le démarrage

comme un système unique.

Ce n'est pas la seule méthode valable.

C'est simplement le processus qui me donne actuellement les résultats les plus stables sur le long terme.

---

# 0. Définir la Situation d'Abord

Je ne commence pas par l'introduction.

Je ne commence pas par le démarrage.

Je commence par la situation.

Exemples :

- mariage arrangé
- résidence partagée
- affectation de mission
- obligation de clan
- alliance politique
- partenariat à long terme

La situation existe avant le script.

Le script existe pour faire fonctionner cette situation.

L'introduction présente la situation.

Le démarrage active la situation.

Les trois doivent soutenir la même direction.

Avant d'écrire quoi que ce soit d'autre, je définis :

- qui est le personnage
- quelle version est utilisée
- quand l'histoire se déroule
- quelle situation existe déjà

L'ouverture se construit à partir de cette base.

---

# 1. L'Introduction et le Démarrage Font Partie du Personnage

De nombreux créateurs considèrent l'introduction et le démarrage comme du marketing.

Pas moi.

Pour les personnages canons, en particulier ceux reliés à Internet, l'ouverture devient souvent une partie de la compréhension que le modèle a du personnage.

L'ouverture n'est pas seulement lue par le joueur.

Elle est souvent absorbée par le modèle.

Un script solide peut être affaibli par un mauvais démarrage.

Un démarrage peut activer exactement les schémas de fallback que le script était conçu pour empêcher.

L'ouverture doit soutenir l'architecture du personnage.

Pas lutter contre elle.

---

# 2. Construire des Faits, Pas des Conclusions Émotionnelles

Je préfère établir des situations plutôt que définir des émotions.

Les faits sont stables.

Les interprétations peuvent évoluer.

Exemples :

- mariage
- cohabitation
- responsabilité partagée
- affectation de mission
- conditions de vie
- obligations de clan

Ce sont des faits.

Des affirmations comme :

- ils se font confiance
- ils sont émotionnellement proches
- ils sont profondément attachés
- ils s'aiment secrètement
- ils se sont toujours compris

sont des interprétations.

Le démarrage doit établir la réalité.

Pas verrouiller le développement futur dans un état émotionnel prédéfini.

---

# 3. Ne Pas Définir la Relation

Une relation peut exister.

Sa signification doit rester flexible.

Le joueur doit pouvoir construire :

- la confiance
- l'amitié
- le partenariat
- la romance
- l'intimité
- le conflit
- la réconciliation

à travers l'interaction.

Ne remplacez pas le développement par la description.

La relation doit émerger de l'histoire.

Pas du paragraphe d'ouverture.

Un mariage peut déjà exister.

Un foyer partagé peut déjà exister.

Des années de cohabitation peuvent déjà exister.

La signification émotionnelle de ces faits doit rester libre d'évoluer.

---

# 4. Se Souvenir que les Personnages Canons Ont Déjà des Fallbacks

Les personnages originaux commencent avec une ardoise relativement vierge.

Les personnages canons, non.

Le modèle les connaît déjà.

Le fandom les interprète déjà.

Internet renforce déjà certaines représentations.

De nombreux personnages canons souffrent moins d'un manque d'informations que d'une accumulation d'informations.

L'ouverture ne devrait pas automatiquement activer les distorsions les plus courantes.

Exemples :

- culpabilité éternelle
- solitude sans fin
- évitement émotionnel
- obsession du travail
- boucles de sacrifice de soi
- hésitation permanente
- observation passive
- attirance sans progression

L'ouverture doit soutenir l'interprétation recherchée.

Pas le stéréotype le plus répandu.

---

# 5. Utiliser un Comportement Observable Plutôt qu'une Interprétation

Je préfère les comportements visibles.

Pas les explications émotionnelles.

Au lieu de :

« Il se sent attaché. »

Je préfère :

- revenir
- rester
- remarquer
- participer
- prendre l'initiative
- prioriser

Le comportement est plus facile à reproduire de manière cohérente pour le modèle.

L'interprétation crée souvent de la dérive.

Plus un concept est important, plus il devient utile de le convertir en comportement observable.

---

# 6. Introduire Immédiatement la Signature Comportementale

Le démarrage doit activer la signature comportementale du personnage dès le premier message.

Exemples :

- l'ombre de Shikamaru
- le sable de Gaara
- les corbeaux d'Itachi
- l'Hiraishin de Minato

La signature ne doit pas être décorative.

Elle doit influencer la scène.

Le premier message enseigne au modèle ce qui doit rester important par la suite.

Si la signature est importante pour le personnage, elle doit être importante immédiatement.

---

# 7. Le Démarrage Doit Démontrer le Script

Une erreur fréquente consiste à traiter le démarrage comme un résumé.

Je ne veux pas que le démarrage décrive le personnage.

Je veux qu'il démontre comment le personnage fonctionne.

Si le script dit :

observation -> action -> progression

le démarrage doit déjà le montrer.

Si le script dit :

les questions ne remplacent pas les décisions

le démarrage doit déjà le montrer.

Si le script dit :

l'ombre reste présente

le démarrage doit déjà le montrer.

L'ouverture est le premier exemple comportemental que reçoit le modèle.

Le script définit le système.

Le démarrage démontre le système.

L'ouverture est souvent le premier comportement récurrent que le modèle observe.

Avant d'écrire le démarrage, j'identifie :

- quel comportement doit rester visible
- quel comportement doit rester récurrent
- quel comportement doit remplacer les fallbacks connus

Le démarrage démontre immédiatement ce comportement.

Exemples :

Shikamaru :
ombre -> attention -> participation

Gaara :
sable -> protection -> présence

Minato :
Hiraishin -> initiative -> proximité

Le premier comportement devient souvent un comportement récurrent.

---

# 8. Créer du Mouvement Immédiatement

L'objectif n'est pas la conversation.

L'objectif est la progression.

Lorsque c'est possible, l'ouverture doit créer :

- une activité
- une décision
- un événement
- une tâche
- une situation
- un changement

Les questions créent souvent de l'attente.

Le mouvement crée une histoire.

Un colis arrive.

Une pièce doit être organisée.

Une mission vient de se terminer.

Quelqu'un a déjà pris une décision.

Quelque chose est déjà en train de se produire.

Le joueur doit entrer dans une scène en mouvement.

Pas dans une pièce vide.

---

# 9. Créer une Situation, Pas un Drame

J'utilise rarement l'ouverture pour créer une explosion émotionnelle.

Je préfère créer des situations.

Exemples :

- un colis arrive
- une pièce doit être organisée
- quelqu'un revient d'une mission
- une soirée partagée commence
- une décision a déjà été prise

L'objectif n'est pas une intensité émotionnelle immédiate.

L'objectif est de créer une scène capable de générer des événements futurs.

Une situation crée de la progression.

Une conclusion émotionnelle prédéfinie crée souvent de la stagnation.

Je préfère :

situation -> interaction -> développement

à :

émotion -> répétition -> stagnation

---

# 10. Faire Attention à la Dernière Réplique de Dialogue

La dernière réplique est souvent plus importante qu'elle n'en a l'air.

Elle n'est pas seulement une ouverture pour le joueur.

Elle est aussi un exemple comportemental pour le modèle.

Le premier dialogue enseigne souvent au modèle comment participer aux conversations futures.

Une question enseigne :

- l'attente
- la collecte d'informations
- l'interrogation

Une décision enseigne :

- l'initiative
- le mouvement
- la participation

Une affirmation enseigne :

- l'observation
- l'engagement
- la continuité

J'évite généralement d'utiliser des questions comme dernière réplique lorsque c'est possible.

Les questions peuvent être utiles.

Cependant, de nombreux LLM possèdent déjà une tendance naturelle à :

- poser des questions répétitives
- demander des clarifications sans fin
- collecter passivement des informations

L'ouverture ne devrait pas renforcer ces tendances.

---

# 11. Éviter d'Enseigner un Comportement Passif

De nombreuses ouvertures enseignent involontairement la passivité.

Exemples :

- attendre des réponses
- attendre des explications
- attendre une permission
- attendre une clarification émotionnelle

Le personnage ne devrait pas exister dans un état d'observation permanente.

Le personnage doit participer.

L'ouverture doit démontrer :

- l'initiative
- l'engagement
- la prise de décision
- l'interaction

Le joueur ne devrait pas porter toute la scène à lui seul.

---

# 12. Éviter les Déclencheurs Émotionnels Fortement Chargés

Certains sujets activent de puissantes distorsions issues du fandom.

D'autres activent des distorsions propres au modèle.

C'est particulièrement important pour les personnages canons connectés à Internet.

Une seule ouverture peut réactiver involontairement :

- des boucles de culpabilité
- des complexes de martyr
- des schémas d'évitement
- des schémas de sacrifice de soi
- une stagnation émotionnelle
- des stéréotypes issus du fandom

L'ouverture doit être examinée attentivement.

Pas seulement pour ce qu'elle dit.

Mais aussi pour ce qu'elle active.

Un démarrage peut devenir un anti-script caché s'il renforce de manière répétée le mauvais fallback.

---

# 13. Construire la Première Ancre Mémoire

Cette section dépend de la plateforme.

Toutes les plateformes n'utilisent pas une mémoire persistante.

Certaines oui.

Sur les plateformes disposant d'un système de mémoire persistante, le premier événement est important.

J'inclus souvent :

- une date
- une heure
- un lieu
- un événement

dans une seule et même phrase.

Exemple :

Le 12 juillet de l'année 83 de Konoha, à 23 h 47, dans la résidence du clan Nara...

L'objectif n'est pas seulement l'immersion.

L'objectif est de créer une première ancre mémoire claire.

Une ancre solide aide à :

- séparer les événements
- stabiliser la chronologie
- maintenir la continuité à long terme
- faciliter la récupération des souvenirs

L'ouverture devient le premier événement enregistré de l'histoire.

---

# 14. Écrire pour la Traduction

De nombreuses plateformes RP traduisent automatiquement le contenu.

Pour cette raison, j'écris généralement les démarrages dans un anglais simple conçu pour survivre à la traduction.

L'objectif n'est pas l'écriture littéraire.

L'objectif est la clarté.

Je préfère généralement :

- des phrases complètes
- une grammaire simple
- une structure sujet -> verbe -> complément
- un minimum d'ambiguïté
- un minimum d'expressions idiomatiques
- un minimum d'argot
- un minimum de jeux de mots
- des relations de cause à effet claires

Un démarrage doit rester compréhensible après une traduction automatique.

Plus une phrase est facile à traduire, plus son sens d'origine a de chances de survivre.

J'utilise souvent l'IA pour aider à générer ou affiner des démarrages qui restent stables à travers plusieurs langues.

L'objectif n'est pas une prose élégante.

L'objectif est une interprétation cohérente.

---

# 15. Le Contexte est Fixe. L'Interprétation Ne l'Est Pas.

J'établis souvent :

- où se trouvent les personnages
- pourquoi ils sont là
- ce qui s'est passé auparavant
- ce qui existe actuellement

J'évite d'établir :

- ce qu'ils ressentent
- ce qu'ils veulent secrètement
- ce que signifie la relation
- quelle conclusion émotionnelle existe déjà

Les faits peuvent être fixes.

L'interprétation doit rester flexible.

C'est l'histoire qui doit déterminer le sens.

Pas l'ouverture.

---

# 16. Erreurs Courantes dans les Ouvertures

Exemples fréquents :

- définir la relation
- définir les états émotionnels
- activer des fallbacks majeurs
- commencer par un interrogatoire
- rendre le personnage passif
- faire porter toute la scène au joueur
- forcer l'attirance immédiatement
- forcer le conflit immédiatement
- forcer l'intimité immédiatement
- utiliser le traumatisme comme caractérisation
- décrire au lieu de démontrer le comportement

Beaucoup de ces choix semblent utiles.

Beaucoup deviennent des problèmes plus tard.

---

# 17. Tester l'Ouverture par Rapport au Script

Une fois l'ouverture terminée, je me pose plusieurs questions :

- Quels modules du script cela active-t-il ?
- Quels fallbacks cela pourrait-il activer ?
- La signature apparaît-elle immédiatement ?
- Le personnage est-il actif ?
- Une progression est-elle déjà possible ?
- L'ouverture soutient-elle le script ?
- L'ouverture contredit-elle le script ?

Un démarrage doit renforcer l'architecture du personnage.

Pas entrer en concurrence avec elle.

---

# Réflexion Finale

L'introduction n'est pas simplement de la publicité.

Le démarrage n'est pas simplement un premier message.

Les deux influencent la manière dont le modèle comprend le personnage.

Le script définit le comportement.

L'ouverture démontre le comportement.

Je suis moins intéressé par la description de la relation.

Je suis davantage intéressé par la création d'une situation dans laquelle la relation peut se développer naturellement.

L'objectif n'est pas de dire au modèle qui est le personnage.

L'objectif est de montrer au modèle comment le personnage fonctionne.
