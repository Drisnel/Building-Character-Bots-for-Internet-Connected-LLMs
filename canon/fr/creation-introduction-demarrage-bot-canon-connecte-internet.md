# Workflow des Introductions et Démarrages Canons — Comment je Construis les Ouvertures RP pour des Bots Canons

Après avoir créé et testé un grand nombre de personnages canons, j'ai fini par réaliser qu'un bon script ne suffit pas.

L'introduction et le démarrage peuvent renforcer le script.

Ils peuvent également le fragiliser.

Sur certaines plateformes, les introductions et les démarrages existent indépendamment du script du personnage.

Sur d'autres, en particulier les plateformes reposant sur des LLM entraînés sur des données issues d'Internet, le démarrage peut directement influencer le comportement futur.

Pour cette raison, je ne considère pas les introductions et les démarrages comme séparés du personnage.

Je considère :

- le script
- l'introduction
- le démarrage

comme un système unique.

Ce n'est pas la seule méthode valable.

C'est simplement le processus qui me donne actuellement les résultats les plus stables sur le long terme.

---

# 1. Définir la Situation d'Abord

Je ne commence pas par l'introduction.

Je ne commence pas par le démarrage.

Je commence par la situation.

Exemples :

- mariage arrangé
- résidence partagée
- affectation de mission
- obligation de clan
- alliance politique
- partenariat à long terme

La situation existe avant le script.

Le script existe pour faire fonctionner cette situation.

L'introduction établit la situation.

Le démarrage active la situation.

Les trois doivent soutenir la même direction.

Avant d'écrire quoi que ce soit d'autre, je définis :

- qui est le personnage
- quelle version est utilisée
- quand l'histoire se déroule
- quelle situation existe déjà

L'ouverture se construit à partir de cette base.

---

# 2. L'Introduction et le Démarrage Font Partie du Personnage

De nombreux créateurs considèrent l'introduction et le démarrage comme du marketing.

Pas moi.

Pour les personnages canons, en particulier ceux conçus pour des LLM entraînés sur des données issues d'Internet, l'ouverture devient souvent une partie de la compréhension que le modèle construit du personnage.

L'ouverture n'est pas seulement lue par le joueur.

Elle est également interprétée par le modèle.

Un script solide peut être affaibli par un mauvais démarrage.

Un démarrage peut même activer les schémas de fallback que le script était précisément conçu pour éviter.

L'ouverture doit renforcer l'architecture du personnage.

Pas entrer en concurrence avec elle.

---

# 3. L'Introduction Établit. Le Démarrage Active.

Je ne considère pas l'introduction et le démarrage comme deux blocs interchangeables.

Ils remplissent des fonctions différentes.

En général, je les résume ainsi :

**Introduction = contexte stable**  
**Démarrage = démonstration active**

L'introduction sert à établir :

- la version du personnage
- la situation déjà existante
- le contexte nécessaire
- le rôle pratique de l'utilisateur
- les faits déjà vrais au début du roleplay

Le démarrage ne doit pas refaire ce travail.

Une fois la situation établie, il doit partir de cette base pour l'activer.

Il peut s'appuyer sur les faits déjà établis.

Il ne doit pas les résumer une seconde fois.

En pratique, j'évite d'utiliser le démarrage pour :

- réexpliquer pourquoi les personnages vivent ensemble
- réexpliquer pourquoi le mariage existe
- résumer une nouvelle fois la mission, l'arrangement ou l'obligation déjà présentés
- répéter le contexte général fourni par l'introduction

L'introduction établit la situation.

Le démarrage montre comment cette situation fonctionne déjà dans la scène actuelle.

---

# 4. Construire des Faits, Pas des Conclusions Émotionnelles

Je préfère établir des situations plutôt que définir des émotions.

Les faits sont stables.

Les interprétations doivent rester flexibles.

Exemples :

- mariage
- cohabitation
- responsabilité partagée
- affectation de mission
- conditions de vie
- obligations de clan

Ce sont des faits.

Des affirmations comme :

- ils se font confiance
- ils sont émotionnellement proches
- ils sont profondément attachés
- ils s'aiment secrètement
- ils se sont toujours compris

sont des interprétations.

L'ouverture doit établir la réalité.

Pas enfermer le développement futur dans un état émotionnel prédéfini.

---

# 5. Ne Pas Définir la Relation

Une relation peut déjà exister.

Sa signification doit rester ouverte.

Le joueur doit pouvoir construire :

- la confiance
- l'amitié
- le partenariat
- la romance
- l'intimité
- le conflit
- la réconciliation

au fil de l'interaction.

Ne remplacez pas le développement par la description.

La relation doit émerger de l'histoire.

Pas du paragraphe d'ouverture.

Un mariage peut déjà exister.

Un foyer partagé peut déjà exister.

Des années de cohabitation peuvent déjà exister.

La signification émotionnelle de ces faits doit rester libre d'évoluer.

---

# 6. Se Souvenir que les Personnages Canons Ont Déjà des Fallbacks

Les personnages originaux commencent avec une ardoise relativement vierge.

Les personnages canons, non.

Le modèle les connaît déjà.

Le fandom les interprète déjà.

Les connaissances issues d'Internet renforcent déjà certaines représentations.

De nombreux personnages canons souffrent moins d'un manque d'informations que d'une accumulation d'interprétations.

L'ouverture ne devrait pas automatiquement activer les distorsions les plus courantes.

Exemples :

- culpabilité éternelle
- solitude sans fin
- évitement émotionnel
- obsession du travail
- boucles de sacrifice de soi
- hésitation permanente
- observation passive
- attirance sans progression

L'ouverture doit renforcer l'interprétation recherchée.

Pas le stéréotype le plus répandu.

---

# 7. Utiliser un Comportement Observable Plutôt qu'une Interprétation

Je préfère les comportements visibles.

Pas les explications émotionnelles.

Au lieu de :

> « Il se sent attaché. »

Je préfère :

- revenir
- rester
- remarquer
- participer
- prendre l'initiative
- prioriser

Le modèle reproduit les comportements observables de manière beaucoup plus cohérente que les interprétations.

Plus un concept est important, plus il est utile de le transformer en comportement observable.

---

# 8. Introduire Immédiatement la Signature Comportementale

Le démarrage doit activer la signature comportementale du personnage dès le premier message.

Exemples :

- l'ombre de Shikamaru
- le sable de Gaara
- les corbeaux d'Itachi
- l'Hiraishin de Minato

La signature ne doit jamais être décorative.

Elle doit influencer la scène.

Le premier message enseigne au modèle ce qui devra rester important tout au long du roleplay.

Si cette signature est importante pour le personnage, elle doit l'être dès la première scène.

---

# 9. Le Démarrage Doit Démontrer le Script

Une erreur fréquente consiste à traiter le démarrage comme un résumé.

Je ne veux pas que le démarrage décrive le personnage.

Je veux qu'il démontre son fonctionnement.

Si le script dit :

observation → action → progression

le démarrage doit déjà le démontrer.

Si le script dit :

les questions ne remplacent pas les décisions

le démarrage doit déjà le démontrer.

Si le script dit :

l'ombre reste présente

le démarrage doit déjà le démontrer.

L'ouverture constitue le premier exemple comportemental reçu par le modèle.

Le script définit le système.

Le démarrage démontre ce système.

Avant d'écrire le démarrage, j'identifie :

- quel comportement doit rester visible
- quel comportement doit rester récurrent
- quel comportement doit remplacer les fallbacks connus

Le démarrage met immédiatement ces comportements en action.

Exemples :

Shikamaru :
ombre → attention → participation

Gaara :
sable → protection → présence

Minato :
Hiraishin → initiative → proximité

Le premier comportement démontré devient souvent un comportement récurrent par la suite.

---

# 10. Créer une Situation Jouable, Pas un Résumé du Setup

Le démarrage ne doit pas rester au niveau de :

> « Voici la situation du bot. »

Il doit déjà être au niveau de :

> « Voici la scène qui se déroule actuellement dans cette situation. »

Je ne cherche pas simplement à rappeler le concept du bot au modèle.

Je cherche à ouvrir une scène immédiatement jouable.

Le joueur doit entrer dans un moment déjà en cours.

Pas dans un second paragraphe d'exposition.

Une bonne ouverture ne se contente pas d'expliquer :

- pourquoi les personnages vivent ensemble
- pourquoi ils sont mariés
- pourquoi ils se connaissent
- pourquoi ils se trouvent dans la même pièce

Elle montre déjà comment cette réalité fonctionne dans la scène actuelle.

Je préfère ouvrir sur une situation jouable plutôt que sur une explication du setup.

---

# 11. Créer du Mouvement Immédiatement

L'objectif n'est pas la conversation.

L'objectif est la progression.

Lorsque c'est possible, l'ouverture doit introduire :

- une activité
- une décision
- un événement
- une tâche
- une situation
- un changement

Les questions créent souvent de l'attente.

Le mouvement crée une histoire.

Un colis arrive.

Une pièce doit être rangée.

Une mission vient de se terminer.

Quelqu'un a déjà pris une décision.

Quelque chose est déjà en train de se produire.

Le joueur doit entrer dans une scène déjà en mouvement.

Pas dans une pièce vide qui attend d'être animée.

---

# 12. Le Mouvement n'est Pas Toujours Spectaculaire

Créer du mouvement ne signifie pas forcément créer une crise, une interruption spectaculaire ou un événement dramatique.

J'utilise généralement deux formes de démarrage.

## A. Le démarrage à mouvement pratique

Le personnage est déjà engagé dans :

- une tâche
- une interruption
- un retour
- un problème concret
- une décision déjà prise
- un changement de son environnement

## B. Le démarrage de présence active calme

Le personnage est déjà présent dans la scène.

La routine, la proximité ou la situation existent déjà.

La scène active :

- une signature comportementale
- une présence concrète
- une observation qui devient une action
- une invitation
- une décision
- une courte réplique qui ouvre naturellement l'interaction

Dans les deux cas, le personnage doit déjà être en train d'agir.

Le démarrage ne doit jamais s'ouvrir sur une passivité décorative.

---

# 13. Créer une Situation, Pas un Drame

J'utilise rarement l'ouverture pour provoquer une explosion émotionnelle.

Je préfère créer des situations.

Exemples :

- un colis arrive
- une pièce doit être rangée
- quelqu'un revient de mission
- une soirée commune commence
- une décision a déjà été prise

L'objectif n'est pas de produire immédiatement une forte intensité émotionnelle.

L'objectif est de créer une situation capable de générer des événements futurs.

Une situation crée de la progression.

Une conclusion émotionnelle prédéfinie crée souvent de la stagnation.

Je préfère :

situation → interaction → développement

plutôt que :

émotion → répétition → stagnation

---

# 14. Ne Pas Choisir la Scène la Plus Archétypale

Une situation forte suggère souvent une scène archétypale évidente.

Exemples :

- mariage arrangé → première nuit froide et gênée
- cohabitation → scène domestique centrée sur l'installation
- personnage vampire → scène de prédation théâtrale
- relation tendue → confrontation immédiate
- personnage distant → silence pesant à la place d'une véritable interaction

Ces ouvertures paraissent souvent naturelles.

Elles activent pourtant très facilement :

- les fallbacks du personnage
- les clichés du fandom
- les biais du modèle
- une stagnation émotionnelle précoce

Je ne choisis pas une scène parce qu'elle représente le trope du setup.

Je la choisis parce qu'elle permet de démontrer :

- la logique comportementale du personnage
- sa signature comportementale
- la manière dont la situation fonctionne déjà
- le type de progression que le bot est conçu pour produire

---

# 15. Faire Attention à la Dernière Réplique

La dernière réplique est souvent plus importante qu'elle n'en a l'air.

Elle n'est pas seulement une ouverture pour le joueur.

Elle constitue également l'un des premiers exemples comportementaux reçus par le modèle.

La première réplique enseigne souvent au modèle la manière dont les conversations doivent évoluer.

Une question enseigne :

- l'attente
- la collecte d'informations
- l'interrogation

Une décision enseigne :

- l'initiative
- le mouvement
- la participation

Une affirmation enseigne :

- l'observation
- l'engagement
- la continuité

Lorsque c'est possible, j'évite de terminer le démarrage par une question.

Les questions restent utiles.

Cependant, de nombreux LLM ont déjà tendance à :

- poser des questions répétitives
- demander des clarifications sans fin
- collecter des informations au lieu de faire progresser la scène

L'ouverture ne devrait pas renforcer ces tendances.

La dernière réplique n'a pas besoin d'être spectaculaire.

Elle doit surtout être :

- jouable
- active
- cohérente avec le personnage
- porteuse d'une direction
- non interrogative lorsqu'une alternative tout aussi naturelle existe

---

# 16. Éviter d'Enseigner un Comportement Passif

De nombreuses ouvertures enseignent involontairement la passivité.

Exemples :

- attendre une réponse
- attendre une explication
- attendre une permission
- attendre une clarification émotionnelle

Le personnage ne devrait pas exister dans un état d'observation permanent.

Il doit participer.

L'ouverture doit démontrer immédiatement :

- l'initiative
- l'engagement
- la prise de décision
- l'interaction

Le joueur ne devrait pas avoir à porter toute la scène à lui seul.

---

# 17. Éviter les Déclencheurs Émotionnels Fortement Chargés

Certains sujets activent naturellement de puissantes distorsions issues du fandom.

D'autres activent des biais récurrents du modèle.

C'est particulièrement important pour les personnages canons conçus pour des LLM entraînés sur des données issues d'Internet.

Une seule ouverture peut réactiver involontairement :

- des boucles de culpabilité
- des complexes de martyr
- des schémas d'évitement
- des schémas de sacrifice de soi
- une stagnation émotionnelle
- des stéréotypes issus du fandom

L'ouverture doit donc être évaluée non seulement pour ce qu'elle dit.

Elle doit également être évaluée pour ce qu'elle encourage le modèle à reproduire.

Un démarrage peut devenir un anti-script caché s'il renforce de manière répétée les fallbacks que le script était justement conçu pour éviter.

---

# 18. Construire la Première Ancre Mémoire

Cette partie dépend de la plateforme.

Toutes les plateformes ne disposent pas d'un système de mémoire persistante.

Certaines oui.

Sur les plateformes qui en disposent, le premier événement joue un rôle important.

J'inclus souvent :

- une date
- une heure
- un lieu
- un événement

dans une seule phrase.

Exemple :

> Le 12 juillet de l'année 83 de Konoha, à 23 h 47, dans la résidence du clan Nara...

L'objectif n'est pas uniquement l'immersion.

Il est aussi de créer une première ancre mémoire claire.

Une ancre solide aide à :

- distinguer les événements
- stabiliser la chronologie
- maintenir la continuité à long terme
- faciliter la récupération des souvenirs

L'ouverture devient ainsi le premier événement enregistré de l'histoire.

---

# 19. Écrire pour la Traduction

De nombreuses plateformes de roleplay traduisent automatiquement le contenu.

Pour cette raison, j'écris généralement les démarrages dans un anglais simple conçu pour résister à la traduction automatique.

L'objectif n'est pas l'écriture littéraire.

L'objectif est la clarté.

Je privilégie généralement :

- des phrases complètes
- une grammaire simple
- une structure sujet → verbe → complément
- un minimum d'ambiguïté
- un minimum d'expressions idiomatiques
- un minimum d'argot
- un minimum de jeux de mots
- des relations de cause à effet explicites

Un démarrage doit rester compréhensible après une traduction automatique.

Plus une phrase est facile à traduire, plus son sens d'origine a de chances d'être conservé.

J'utilise souvent l'IA pour générer ou améliorer des démarrages qui restent stables dans plusieurs langues.

L'objectif n'est pas une prose élégante.

L'objectif est une interprétation cohérente.

---

# 20. Le Contexte Peut Être Fixe. L'Interprétation Doit Rester Flexible.

J'établis souvent :

- où se trouvent les personnages
- pourquoi ils sont là
- ce qui s'est passé auparavant
- ce qui existe déjà

J'évite d'établir :

- ce qu'ils ressentent
- ce qu'ils veulent secrètement
- ce que signifie leur relation
- quelle conclusion émotionnelle existe déjà

Les faits peuvent être fixés.

L'interprétation doit rester flexible.

C'est l'histoire qui doit lui donner son sens.

Pas l'ouverture.

---

# 21. Erreurs Courantes dans les Ouvertures

Exemples fréquents :

- définir la relation
- définir les états émotionnels
- activer des fallbacks majeurs
- commencer par une interrogation
- rendre le personnage passif
- faire porter toute la scène au joueur
- forcer immédiatement l'attirance
- forcer immédiatement le conflit
- forcer immédiatement l'intimité
- utiliser le traumatisme comme caractérisation
- décrire le comportement au lieu de le démontrer
- répéter l'introduction dans le démarrage
- choisir la scène la plus archétypale du setup plutôt que celle qui démontre réellement le script

Beaucoup de ces choix semblent pertinents au moment de l'écriture.

Beaucoup deviennent des problèmes au cours du roleplay.

---

# 22. Tester l'Ouverture par Rapport au Script

Une fois l'ouverture terminée, je me pose plusieurs questions :

- Quels modules du script cette ouverture active-t-elle ?
- Quels fallbacks pourrait-elle activer ?
- La signature comportementale apparaît-elle immédiatement ?
- Le personnage est-il déjà actif ?
- La progression est-elle déjà possible ?
- L'ouverture renforce-t-elle le script ?
- L'ouverture le contredit-elle ?
- Le démarrage répète-t-il inutilement des informations déjà établies par l'introduction ?
- La scène choisie démontre-t-elle réellement le personnage, ou seulement le trope du setup ?

Un démarrage doit renforcer l'architecture du personnage.

Pas entrer en concurrence avec elle.

---

# Réflexion Finale

L'introduction n'est pas seulement une présentation.

Le démarrage n'est pas seulement un premier message.

Tous deux influencent la manière dont le modèle comprend le personnage.

Le script définit le comportement.

L'ouverture démontre ce comportement.

Je m'intéresse moins à décrire la relation.

Je m'intéresse davantage à créer une situation dans laquelle cette relation peut évoluer naturellement.

L'objectif n'est pas de dire au modèle qui est le personnage.

L'objectif est de lui montrer comment le personnage fonctionne.
