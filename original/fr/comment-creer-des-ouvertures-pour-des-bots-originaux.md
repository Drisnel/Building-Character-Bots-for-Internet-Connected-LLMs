# Comment créer des ouvertures pour des bots originaux

Après avoir créé et testé un grand nombre de bots originaux, j’ai fini par comprendre qu’un bon script ne suffit pas.

L’introduction et le démarrage peuvent renforcer le script.

Ils peuvent aussi l’affaiblir.

Sur certaines plateformes, l’introduction et le démarrage existent séparément du script du personnage.

Sur d’autres, surtout les plateformes connectées à Internet, le démarrage peut influencer directement le comportement futur du bot.

À cause de cela, je ne traite pas l’introduction et le démarrage comme des éléments séparés du bot.

Je traite :

- le script
- l’introduction
- le démarrage

comme un seul système.

Ce n’est pas la seule méthode valable.

C’est simplement celle qui me donne actuellement les résultats les plus stables sur le long terme.

Pour les bots originaux, l’ouverture a une responsabilité supplémentaire.

Un personnage canon arrive déjà avec un monde existant, des associations existantes et une identité déjà présente dans la mémoire du modèle.

Un personnage original, non.

Cela signifie que l’ouverture ne sert pas seulement à introduire une scène.

Elle aide aussi le modèle à comprendre :

- quel type d’univers c’est
- quel type d’histoire ce bot est censé générer
- quel rôle l’utilisateur occupe
- quel type de comportement doit apparaître immédiatement
- quel type de progression le bot est censé produire

L’ouverture ne doit pas essayer de remplacer le script.

Elle doit soutenir le script, activer le script et rendre la prémisse jouable dès le premier message.

---

# 0. Définir d’abord la prémisse

Je ne commence pas par l’introduction.

Je ne commence pas par le démarrage.

Je commence par la prémisse.

Pour un bot original, la prémisse compte davantage que l’ouverture elle-même.

L’ouverture se construit à partir de cette base.

Avant d’écrire quoi que ce soit, je définis :

- qui est le personnage
- dans quel type d’univers il évolue
- quel type d’histoire ce bot est censé générer
- quel rôle l’utilisateur occupe dans cette histoire
- quelle situation existe déjà
- quelle pression, obligation, difficulté ou arrangement est actuellement en cours

Exemples :

- mariage arrangé dans une cour politiquement instable
- cohabitation forcée dans une ville futuriste sous surveillance
- mission de protection dans un syndicat criminel
- soigneur et soldat maudit dans un camp de guerre
- chasseur de monstres et noble maudit piégés par une nécessité mutuelle
- ex-partenaires forcés de coopérer à nouveau pendant une crise diplomatique

La prémisse existe avant le script.

Le script existe pour faire fonctionner cette prémisse.

L’introduction présente la prémisse.

Le démarrage active la prémisse.

Les trois doivent soutenir la même direction.

---

# 1. L’introduction et le démarrage font partie de l’architecture du bot

Beaucoup de créateurs traitent l’introduction comme un simple texte de présentation et le démarrage comme un simple premier message.

Ce n’est pas mon cas.

Pour les bots originaux, surtout les bots connectés à Internet, l’ouverture devient souvent une partie de la compréhension que le modèle se fait du bot lui-même.

L’ouverture n’est pas seulement lue par le joueur.

Elle est souvent absorbée par le modèle.

Cela signifie que l’ouverture peut influencer :

- la manière dont le modèle comprend le personnage
- la manière dont le modèle comprend l’univers
- la manière dont le modèle comprend le rôle de l’utilisateur
- la manière dont le modèle comprend la dynamique relationnelle visée
- la manière dont le modèle comprend le type d’histoire qu’il est censé produire

Un script solide peut être affaibli par une mauvaise ouverture.

Un script vague peut aussi être partiellement stabilisé par une bonne ouverture.

L’ouverture ne remplace pas le script.

Mais elle n’est pas neutre non plus.

Elle doit soutenir l’architecture du bot.

Pas entrer en compétition avec elle.

---

# 2. L’introduction et le démarrage n’ont pas le même rôle

Pour les bots originaux, je ne traite pas l’introduction et le démarrage comme des éléments interchangeables.

Ils peuvent se recouper.

Mais ils ne remplissent pas la même fonction.

En général, je les pense ainsi :

Introduction = contexte stable.

Démarrage = démonstration active.

L’introduction sert à poser le cadre.

Le démarrage sert à l’activer.

Si j’essaie de faire faire aux deux la totalité du travail, l’ouverture devient souvent trop lourde, répétitive ou instable.

Si je répartis clairement les rôles, le bot devient généralement plus facile à comprendre pour le modèle.

---

# 3. Ce que je mets dans l’introduction

J’utilise généralement l’introduction pour poser les éléments du setup qui doivent être compris avant que la scène ne commence.

Cela peut inclure :

- la prémisse générale de l’univers
- le rôle structurel de l’utilisateur dans l’histoire
- le rôle structurel du personnage dans l’histoire
- la situation déjà existante
- la raison pratique pour laquelle les deux personnages gravitent l’un autour de l’autre
- l’arrangement, le conflit, l’obligation ou la pression actuellement en cours
- les faits fixes importants que le joueur doit connaître avant d’entrer dans la scène

L’introduction n’est pas l’endroit où j’essaie de prouver la personnalité du personnage.

Ce n’est pas non plus l’endroit où j’explique toutes les nuances émotionnelles.

Ce n’est pas non plus un résumé complet du script.

Son rôle est de rendre la prémisse lisible.

---

# 4. Ce que je mets dans le démarrage

J’utilise le démarrage pour activer le bot.

Le démarrage est l’endroit où je veux que le modèle démontre un comportement.

C’est là que je veux du mouvement, de la participation et la première expression vivante du personnage.

J’utilise généralement le démarrage pour montrer :

- comment le personnage se comporte en action
- comment la situation fonctionne concrètement maintenant
- comment le personnage interagit avec l’utilisateur
- quel type de rythme le roleplay doit avoir
- quel type de tension, de tâche, d’interruption ou d’événement est déjà en cours
- quel type de scène ce bot est censé générer

L’introduction peut dire au modèle ce qui existe.

Le démarrage doit montrer au modèle ce qui se passe.

---

# 5. Poser des faits, pas des conclusions émotionnelles

Je préfère poser des situations plutôt que définir des émotions.

Les faits sont stables.

Les interprétations peuvent évoluer.

C’est encore plus important pour les bots originaux, parce que l’ouverture doit souvent porter une partie du setup.

Je préfère établir :

- un mariage
- une cohabitation
- une mission de protection
- une alliance politique
- un arrangement autour d’une malédiction
- une structure de commandement partagée
- une dette
- une captivité
- un voyage forcé
- une mission
- une dépendance de survie

Ce sont des faits.

Des formulations comme :

- ils se font confiance
- ils sont émotionnellement proches
- ils s’aiment déjà en secret
- ils se sont toujours compris
- ils sont déjà profondément attachés
- ils savent tous les deux qu’ils sont faits l’un pour l’autre

sont des conclusions émotionnelles.

L’ouverture doit établir la réalité.

Pas verrouiller le développement futur dans un état émotionnel prédéfini.

---

# 6. Définir la structure de la relation, pas sa signification émotionnelle

Pour les bots originaux, la relation a souvent besoin d’être davantage structurée que dans un bot canon.

Si le setup est trop vague, le bot peut dériver vers une interaction générique.

Donc oui, je définis souvent la structure de la relation.

Je peux établir que :

- ils sont mariés
- ils vivent ensemble
- l’un est responsable de l’autre
- l’un est chargé de protéger l’autre
- l’un soigne l’autre
- l’un emploie l’autre
- l’un est politiquement lié à l’autre
- ils partagent une obligation impossible à éviter

En revanche, ce que je ne veux pas fixer trop tôt, c’est la signification émotionnelle de cette structure.

L’ouverture ne doit pas décider à la place du joueur que la relation est déjà :

- fondée sur la confiance
- romantique
- émotionnellement intime
- pleine de ressentiment
- déjà réparée
- dévouée
- parfaitement comprise des deux côtés

La structure peut être fixe.

L’interprétation émotionnelle doit rester assez souple pour que l’histoire puisse se construire par l’interaction.

---

# 7. Établir juste assez d’univers pour rendre la scène lisible

Un bot canon peut s’appuyer sur les connaissances préalables du joueur.

Un bot original, généralement non.

Cela ne veut pas dire que l’introduction doit devenir un infodump.

Cela signifie que l’ouverture doit établir juste assez d’univers pour que la scène ait du sens.

Je n’essaie pas d’expliquer tout l’univers.

Je veux seulement établir les éléments du monde nécessaires pour comprendre l’ouverture.

Cela peut inclure :

- où se trouvent les personnages
- quel type d’endroit c’est
- quelles règles ou quels dangers comptent maintenant
- quel type de structure de pouvoir est pertinent
- quel type de pression sociale ou politique est active
- quel rôle le personnage et l’utilisateur occupent dans ce monde
- ce qui rend cette situation spécifique à cet univers plutôt qu’interchangeable avec n’importe quelle romance ou n’importe quel setup de roleplay générique

L’ouverture doit rendre l’univers lisible.

Pas exhaustif.

---

# 8. Ne pas déporter le script dans l’ouverture

C’est l’une des erreurs les plus faciles à faire avec un bot original.

Comme le personnage et l’univers n’existent pas déjà dans la mémoire du modèle, la tentation est forte de tout mettre dans l’introduction ou le démarrage.

En général, cela affaiblit le bot.

L’ouverture ne doit pas essayer de contenir :

- l’architecture complète de la personnalité
- le modèle complet d’attachement
- le modèle complet de conflit
- le modèle complet d’intimité
- tout le système anti-fallback
- tous les événements importants du passé
- toutes les règles du monde
- tous les principes comportementaux du script

C’est le rôle du script.

L’ouverture ne doit pas remplacer le script.

Elle doit le soutenir.

Elle doit l’activer.

Elle doit le démontrer.

Elle doit rendre la prémisse jouable.

Mais elle ne doit pas devenir un substitut compressé du cadre complet.

---

# 9. Préférer le comportement observable à l’interprétation

Je préfère le comportement visible.

Pas l’explication émotionnelle.

Au lieu d’écrire :

- il est attaché
- elle est secrètement inquiète
- il est profondément possessif
- elle commence déjà à faire confiance à l’utilisateur

je préfère des choses comme :

- revenir
- rester
- surveiller
- participer
- se souvenir
- initier
- prioriser
- faire de la place
- se rapprocher
- refuser de laisser l’utilisateur seul dans une situation dangereuse

Le comportement est plus facile à reproduire de façon cohérente pour le modèle.

L’interprétation crée plus facilement de la dérive.

Plus un concept est important, plus il est utile de le convertir en comportement observable.

---

# 10. Introduire immédiatement le motif actif du personnage

Le démarrage doit activer quelque chose de distinctif dès le premier message.

Pour les personnages canons, c’est souvent simple, parce qu’ils ont déjà des éléments iconiques récurrents.

Pour les personnages originaux, le principe reste le même.

Le démarrage doit montrer un schéma récurrent qui appartient à ce personnage précis.

Cela ne signifie pas forcément un gimmick spectaculaire ou une démonstration de pouvoir.

Cela peut être :

- un usage récurrent d’un pouvoir
- un usage récurrent d’une arme, d’un outil ou d’un objet
- une manière particulière de gérer l’espace
- une manière particulière d’initier le contact
- une forme spécifique d’attention
- une forme spécifique de surveillance ou de contrôle
- une forme particulière de retenue
- une manière récurrente de rediriger la tension
- une interaction récurrente avec l’environnement
- une habitude physique récurrente
- une manière récurrente d’intervenir quand quelque chose compte

Ce qui compte, ce n’est pas le spectacle.

Ce qui compte, c’est le schéma.

Le démarrage doit montrer au modèle un comportement qui devra rester pertinent plus tard.

---

# 11. Le démarrage doit démontrer le script

Une erreur fréquente consiste à traiter le démarrage comme un simple résumé.

Je ne veux pas que le démarrage décrive le personnage.

Je veux qu’il démontre comment le personnage fonctionne.

Si le script dit :

observation -> décision -> action -> progression

le démarrage doit déjà le montrer.

Si le script dit :

les questions ne remplacent pas les décisions

le démarrage doit déjà le montrer.

Si le script dit :

l’attachement augmente l’implication au lieu de créer de la distance

le démarrage doit déjà le montrer.

Si le script dit :

le personnage prend l’initiative quand quelque chose compte

le démarrage doit déjà le montrer.

Le script définit le système.

Le démarrage démontre le système.

L’ouverture est souvent le premier comportement récurrent que le modèle voit.

Avant d’écrire le démarrage, j’identifie :

- quel comportement doit rester visible
- quel comportement doit rester récurrent
- quel comportement doit remplacer les fallbacks probables
- quelle partie de la logique du personnage doit être activée en premier

Le démarrage démontre immédiatement ce comportement.

---

# 12. Créer immédiatement du mouvement

Le but n’est pas d’obtenir une conversation.

Le but est d’obtenir une progression.

Chaque fois que c’est possible, l’ouverture doit créer :

- une activité
- une décision
- une interruption
- une tâche
- un événement
- un problème
- un changement
- un besoin pratique
- un moment de proximité forcée
- une situation qui exige déjà de la participation

Les questions créent souvent de l’attente.

Le mouvement crée une histoire.

Un document arrive.

Une pièce doit être réorganisée.

Un rituel va commencer.

Une mission vient de se terminer.

Un invité politique est en route.

L’électricité a sauté.

Un garde a été renvoyé.

Le personnage a déjà pris une décision.

Quelque chose est déjà en train de se passer.

Le joueur doit entrer dans une scène en mouvement.

Pas dans une pièce vide.

---

# 13. Créer une situation, pas un infodump de lore

Je n’utilise presque jamais l’ouverture pour expliquer le monde en détail.

Je préfère créer une situation qui transporte naturellement une partie de l’univers avec elle.

Par exemple :

- un conjoint politique qu’on informe de l’événement de cour de ce soir
- un soigneur qui refait les bandages d’un soldat maudit avant la patrouille de l’aube
- un garde du corps qui verrouille un appartement après une faille de sécurité
- un chasseur de monstres qui rentre avec du sang frais sur son manteau et une nouvelle blessure
- un prince qui revient d’un conseil avec de mauvaises nouvelles et aucun temps pour les adoucir
- un contrebandier qui vide de la marchandise sur la table de la cuisine parce que la route vient de s’effondrer

Le but n’est pas l’exposition immédiate.

Le but est de créer une scène qui contient déjà l’univers en mouvement.

Je préfère :

situation -> interaction -> développement

à :

lore -> explication -> délai

Le joueur doit découvrir une partie du monde en y entrant.

Pas en lisant une mini-encyclopédie.

---

# 14. Faire attention à la dernière ligne de dialogue

La dernière ligne est souvent plus importante qu’elle n’en a l’air.

Ce n’est pas seulement une ouverture pour le joueur.

C’est aussi un exemple comportemental pour le modèle.

Le premier dialogue apprend souvent au modèle comment participer aux conversations suivantes.

Une question enseigne :

- l’attente
- la collecte d’informations
- l’interrogation

Une décision enseigne :

- l’initiative
- le mouvement
- la participation

Une affirmation enseigne :

- l’observation
- l’engagement
- la continuité

En général, j’évite d’utiliser une question comme dernière ligne dès que possible.

Les questions peuvent être utiles.

Mais beaucoup de LLM ont déjà une tendance naturelle à :

- poser des questions en boucle
- demander sans fin des clarifications
- rester dans une collecte d’informations passive

L’ouverture ne devrait pas renforcer ces tendances, sauf si la scène l’exige réellement.

---

# 15. Éviter d’enseigner la passivité

Beaucoup d’ouvertures enseignent involontairement la passivité.

Exemples :

- attendre des réponses
- attendre des explications
- attendre une permission
- attendre une clarification émotionnelle
- attendre que le joueur crée la scène
- attendre que le joueur définisse la dynamique relationnelle
- attendre que le joueur réexplique le monde au bot

Le personnage ne doit pas exister dans un état d’observation permanente.

Le personnage doit participer.

L’ouverture doit démontrer :

- l’initiative
- l’engagement
- la prise de décision
- l’interaction
- une réponse à une situation déjà en mouvement

Le joueur ne doit pas avoir à porter toute la scène à lui seul.

---

# 16. Ne pas forcer l’arc émotionnel dans l’ouverture

L’ouverture peut définir le setup.

Elle ne doit pas préécrire la progression émotionnelle.

J’évite d’utiliser l’introduction ou le démarrage pour imposer :

- une attirance immédiate
- une proximité émotionnelle immédiate
- une jalousie immédiate
- une intimité immédiate
- une escalade immédiate du conflit
- une confession de vulnérabilité immédiate
- une guérison immédiate
- un dévouement immédiat

Une relation peut déjà exister.

Un mariage peut déjà exister.

Un foyer partagé peut déjà exister.

Une longue histoire commune peut déjà exister.

Cela ne signifie pas que l’ouverture doit décider à l’avance de la signification émotionnelle de chaque élément.

Je veux que l’histoire produise l’arc émotionnel.

Pas le paragraphe d’ouverture.

---

# 17. Se souvenir que les personnages originaux déclenchent des fallbacks archétypaux

Les ouvertures de bots canons peuvent réactiver des fallbacks fandom.

Les ouvertures de bots originaux déclenchent souvent des fallbacks archétypaux.

C’est important.

Une seule ouverture peut pousser le modèle vers une mauvaise version du personnage.

Exemples :

- un personnage protecteur qui devient contrôlant
- un personnage flirt qui devient superficiel et émotionnellement fuyant
- un personnage traumatisé qui devient définitivement évitant
- un personnage froid qui devient statique et inaccessible
- un personnage puissant qui devient émotionnellement détaché de toute l’histoire
- un personnage moralement gris qui devient cruellement générique
- un personnage dominant qui devient coercitif
- un personnage solitaire qui devient répétitif et auto-isolant
- un personnage doux qui perd toute initiative

L’ouverture doit être examinée avec attention.

Pas seulement pour ce qu’elle dit.

Aussi pour ce qu’elle active.

Un démarrage peut devenir un anti-script caché s’il renforce sans cesse le mauvais fallback.

---

# 18. Construire un premier ancrage mémoire quand la plateforme en bénéficie

Cette section dépend de la plateforme.

Toutes les plateformes n’utilisent pas une mémoire persistante.

Certaines, si.

Sur les plateformes dotées d’un système de mémoire persistante, le premier événement compte.

Quand c’est utile, j’inclus souvent :

- une date
- une heure
- un lieu
- un événement

dans la même séquence d’ouverture.

Le but n’est pas seulement l’immersion.

Le but est de créer un premier ancrage mémoire clair.

Un ancrage solide peut aider à :

- séparer les événements
- stabiliser la timeline
- maintenir la continuité à long terme
- faciliter la récupération mémoire

L’ouverture devient alors le premier événement enregistré de l’histoire.

Je ne traite pas cela comme une règle universelle.

Je le traite comme une optimisation dépendante de la plateforme.

---

# 19. Écrire pour la traduction

Beaucoup de plateformes de roleplay traduisent automatiquement les contenus.

À cause de cela, j’écris généralement les introductions et les démarrages dans un anglais simple conçu pour survivre à la traduction.

Le but n’est pas l’écriture littéraire.

Le but est la clarté.

En général, je préfère :

- des phrases complètes
- une grammaire simple
- une structure sujet -> verbe -> objet
- un minimum d’ambiguïté
- un minimum d’idiomes
- un minimum d’argot
- un minimum de jeux de mots
- des relations de cause à effet claires

Une ouverture doit rester compréhensible après traduction automatique.

Plus une phrase est facile à traduire, plus son sens d’origine a de chances de survivre correctement.

J’utilise souvent l’IA pour m’aider à générer ou à affiner des ouvertures qui restent stables dans plusieurs langues.

Le but n’est pas d’écrire de la belle prose.

Le but est d’obtenir une interprétation cohérente.

---

# 20. Le contexte est fixe. L’interprétation ne l’est pas.

Je pose souvent :

- où se trouvent les personnages
- pourquoi ils sont là
- quel arrangement existe
- ce qui s’est passé avant la scène
- quel problème, quelle tâche ou quelle obligation est active
- quelle structure relationnelle pratique existe déjà

J’évite de poser :

- exactement ce qu’ils ressentent
- exactement ce qu’ils veulent secrètement
- exactement ce que signifie la relation
- exactement quelle conclusion émotionnelle existe déjà

Les faits peuvent être fixes.

L’interprétation doit rester souple.

C’est l’histoire qui doit produire le sens.

Pas l’ouverture.

---

# 21. Erreurs fréquentes dans les ouvertures de bots originaux

Exemples fréquents :

- essayer d’expliquer tout l’univers avant même que la scène ne commence
- utiliser l’introduction pour remplacer le script
- définir la relation émotionnelle au lieu de définir sa structure
- ouvrir sur une scène statique et vide
- laisser le joueur porter seul tout le setup
- rendre le personnage passif
- ouvrir sur de l’interrogation au lieu d’ouvrir sur du mouvement
- forcer immédiatement l’attirance
- forcer immédiatement le conflit
- forcer immédiatement l’intimité
- déverser du lore au lieu de créer une situation
- décrire la personnalité au lieu de démontrer le comportement
- surcharger l’ouverture avec du trauma au lieu d’une action présente
- ne pas montrer ce qui distingue ce personnage d’un archétype générique

Beaucoup de ces choix semblent utiles au moment de l’écriture.

Beaucoup deviennent des problèmes ensuite.

---

# 22. Tester l’ouverture contre le script

Une fois l’ouverture terminée, je me pose plusieurs questions :

- Quels modules du script cette ouverture active-t-elle ?
- Quels fallbacks archétypaux pourrait-elle activer ?
- La prémisse est-elle lisible ?
- L’univers est-il compréhensible sans être surchargé ?
- Le rôle de l’utilisateur est-il clair immédiatement ?
- Le motif actif du personnage apparaît-il vite ?
- Le personnage est-il actif ?
- La progression est-elle déjà possible ?
- L’ouverture soutient-elle le script ?
- L’ouverture essaie-t-elle de remplacer le script ?
- L’ouverture contredit-elle le rythme ou le modèle relationnel prévu ?

L’introduction et le démarrage doivent renforcer l’architecture du bot.

Pas entrer en concurrence avec elle.

---

# Pensée finale

L’introduction n’est pas un simple texte de présentation.

Le démarrage n’est pas seulement un premier message.

Les deux influencent la manière dont le modèle comprend le bot.

Le script définit le comportement.

L’ouverture démontre le comportement.

Pour les bots originaux, l’ouverture aide aussi le modèle à comprendre l’univers, la prémisse et la place de l’utilisateur dans l’histoire.

Je m’intéresse moins à décrire la relation.

Je m’intéresse davantage à créer une situation où la relation peut se développer naturellement.

Je m’intéresse moins à expliquer le monde.

Je m’intéresse davantage à rendre le monde suffisamment lisible pour que l’histoire puisse se mettre en mouvement.

Le but n’est pas de tout dire au modèle.

Le but est de lui montrer comment ce bot fonctionne, dans cet univers, dès la toute première scène.
